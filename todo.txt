## Event backfill (deferred)

The /events feed is capped at ~300 events / 90 days, so the `events` table is only
complete going FORWARD from when polling starts. To get history:

[ ] Timeline API backfill (preferred path — works on PRIVATE repos)
    - GET /repos/{o}/{r}/issues/{n}/timeline  → full, UNCAPPED per-issue/PR history.
    - Gives the meta-events we can't reconstruct: labeled/unlabeled, assigned,
      renamed, review_requested, cross-referenced (issue<->PR<->commit graph over
      time), closed/reopened, merged, head_ref_force_pushed, committed, reviewed.
    - Only needs READ access (no admin). Works on any readable repo, public or private.
    - Cost: one paginated request per PR/issue (fan-out). Mitigate:
        * concurrent fetch (buffer_unordered, like jobs/checks),
        * reuse the PR/issue list we already iterate,
        * incremental via the existing watermark — after the one-time backfill,
          only re-fetch timelines of subjects that changed.
    - Impl notes: octocrab has no typed timeline method → raw client._get + serde,
      JSON payload column (heterogeneous events). Store in a `timeline_events` table
      (or tag rows in `events` with source='timeline'). Set the mockingbird-preview
      Accept header defensively (GA now, but harmless).

[ ] (cheap, free) Synthesized `activity` view
    - A SQL view UNION-ing timestamps we already fully backfill: PR opened/merged/
      closed, review submitted, comment created, commit authored, workflow run.
    - No API calls. Covers CONTENT activity (not label/assign history, not push/star).
    - Nice as: activity = real `events` (forward) UNION synthesized (historical).

REJECTED: GH Archive (gharchive.org / BigQuery firehose).
    - It's the only source for the raw historical firehose (incl. push/star/fork back
      to 2011) and is DuckDB-readable, BUT it is PUBLIC-repos-only. We want private
      repo support, so this is out. Use the Timeline API instead.

## Packaging / features (before publishing)

[ ] Gate Postgres behind a non-default `postgres` cargo feature (pulls in `postgres` +
    `native-tls`). The only feature-gate that buys anything: native-tls -> openssl on Linux
    is the real transitive-dep pain (musl / cross-compile / manylinux / minimal containers).
    Leave DuckDB + SQLite unconditional — self-contained bundled C libs, no system deps, and
    DuckDB is load-bearing anyway (the engine's write path).
    - Touch points: SinkTarget::Postgres variant, build_sink, extract_postgres, rebuild_from_store,
      the PG migrations include, and the binding SinkTarget enums — all need #[cfg(feature)].

[ ] How it interacts with the multi-language bindings (the subtle part):
    - Feature flags are a RUST BUILD-TIME choice. They cleanly benefit Rust *library* consumers
      (they compile entl-core themselves and can drop openssl).
    - The prebuilt bindings (@entl/node .node, the `entl` Python wheel) BAKE IN whatever features
      the publisher enabled at build time. A JS/Python user installing a prebuilt artifact CANNOT
      toggle them — they get whatever was compiled. So the published bindings default to "all sinks
      on"; the feature only helps someone building the binding from source (maturin/napi --features).
    - Consequence: the openssl cost doesn't vanish for the bindings — it just moves to the WHEEL /
      addon build (manylinux, or the openssl-sys `vendored` feature to static-link it), unless you
      also publish a slim no-postgres variant. Decide: vendor openssl in the wheel, or ship variants.
